## Introduction

Every Polkadot validator runs a single binary that hosts dozens of independent components: candidate validation, approval voting, backing, dispute coordination, network bridge, runtime API access, availability storage, and more. None of them talk to each other directly. They all talk to a central **Overseer** that routes their messages and broadcasts global signals (such as "a new relay chain block was imported").

This document walks the Overseer end-to-end at the source-code level:

- **Where it lives** in the codebase and how it's actually generated by a procedural macro.
- **How the outside world reaches it** — the Overseer is purely in-process, so anything from the network or the database arrives via the **Substrate Client** layer underneath.
- **How it's wired up** during node startup, including which `Arc<Client>` clone goes where and which channels are passed to which subsystem.
- **What its main loop actually does** — the three event sources it multiplexes and the five event variants it processes.
- **How a real flow uses it** — we use the PVF precompilation path in `candidate-validation` (the path touched by [PR #11920](https://github.com/paritytech/polkadot-sdk/pull/11920)) as a concrete example: the Overseer is what makes that subsystem wake up once per relay chain block.

**Key files referenced:**

- `polkadot/node/overseer/src/lib.rs` — the Overseer struct, the `#[orchestra]` declaration, the main loop.
- `polkadot/node/subsystem-types/src/lib.rs` — `OverseerSignal`, `ActivatedLeaf`, `ActiveLeavesUpdate`.
- `polkadot/node/subsystem-types/src/messages.rs` — every message type that travels between subsystems.
- `polkadot/node/service/src/builder/mod.rs` — where the Overseer is constructed and wired.
- `polkadot/node/service/src/builder/partial.rs` — where the Substrate Client is instantiated.
- `polkadot/node/core/candidate-validation/src/lib.rs` — example subsystem; its loop is the receive-side counterpart of the Overseer's broadcast.

# Part I: What the Overseer Is

## 1. The Mental Model: Conductor and Musicians

Think of the validator process as an **orchestra**:

- The **Overseer** is the conductor. It does not play music itself; it tells the musicians when to start, stop, and synchronize.
- Each **Subsystem** is a musician. Each one does one specific job: validating candidates, distributing approval votes, holding the availability store, etc.
- Musicians **never look at each other directly**. They watch the conductor. Anything one subsystem needs to tell another goes through the Overseer.

Concretely, this means the Overseer owns one mpsc channel per subsystem. When subsystem A wants to ask subsystem B for something, A sends a message into the Overseer's event channel, and the Overseer pushes it onto B's incoming channel. B never has a direct reference to A.

This is enforced by the type system through the `#[subsystem]` attribute (Section 6): each subsystem declares which message types it can receive and which it is allowed to send. A subsystem cannot send a message it didn't declare; the macro refuses to compile.

## 2. Internal vs External: The Overseer Does Not Talk to the Network

A common misconception is that the Overseer "receives notifications from outside the node." It does not.

The Overseer is **100% in-process**. It owns no sockets. It does not know about peers. It does not deserialize anything from the wire.

The data that comes from outside the node enters through other layers and only **then** reaches the Overseer, already digested:

```
┌───────────────────────────────────────────────────────────────────┐
│                     POLKADOT NODE PROCESS                         │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │ NETWORKING (libp2p)                                     │◄─────┼── peers
│  │   block gossip · sync · transactions · request/response │      │   (TCP/QUIC)
│  └────────────────────────┬────────────────────────────────┘      │
│                           │                                       │
│                           ▼                                       │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │ SUBSTRATE CLIENT (sc-service, sc-consensus, etc.)       │      │
│  │   imports blocks · executes runtime · maintains DB      │      │
│  │   exposes streams (import_notification_stream, etc.)    │      │
│  └────────────────────────┬────────────────────────────────┘      │
│                           │                                       │
│                           │  forward_events()                     │
│                           ▼                                       │
│  ┌─────────────────────────────────────────────────────────┐      │
│  │ OVERSEER  ◄── pure in-process router                    │      │
│  │   broadcasts OverseerSignal::ActiveLeaves to all        │      │
│  │   subsystems · routes messages between them             │      │
│  └────────────────────────┬────────────────────────────────┘      │
│                           │                                       │
│                  ┌────────┴────────┬──────────────┬──── ...       │
│                  ▼                 ▼              ▼               │
│          candidate-          approval-       backing              │
│          validation          voting                               │
│                                                                   │
│  Some subsystems (network-bridge, collator-protocol) DO talk      │
│  to peers, but they do so themselves. The Overseer does not.      │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

The takeaway: when you read code that says "the Overseer receives an `ActiveLeaves`", what actually happened is:

1. A peer gossiped a new block.
2. The networking layer fed it to the Substrate Client.
3. The Client validated and imported it, then emitted on its `import_notification_stream`.
4. A background task (`forward_events`) listening on that stream called `handle.block_imported(...)` on the Overseer's public handle.
5. The Overseer's main loop, blocked in `select!`, dequeued that event and turned it into an `ActiveLeaves` broadcast.

## 3. What the Overseer Is *Not*

To eliminate other common confusions:

- **It is not the entry point of a runtime call.** Runtime calls (`runtime_api().validate_transaction(...)`) go through the `runtime-api` subsystem, which itself dispatches to the Substrate Client's `RuntimeApi` adapter. The Overseer just routes the request message.
- **It is not a thread pool.** Each subsystem owns its own task. The Overseer can spawn jobs *on behalf of* subsystems through `ToOrchestra::SpawnJob` (Section 14), but it doesn't manage thread parallelism.
- **It is not a database or storage layer.** Anything related to chain state lives in the Substrate Client. The Overseer's `active_leaves` field is just an in-memory `HashMap<Hash, BlockNumber>` of currently-active fork tips.

# Part II: Where the Code Lives

## 4. The Crate Layout

The Overseer lives in `polkadot/node/overseer/`:

```
polkadot/node/overseer/
├── Cargo.toml
├── examples/
│   └── minimal-example.rs       ← didactic example of how to build one
└── src/

    ├── metrics.rs                     ← Prometheus metrics
    ├── memory_stats.rs                ← memory tracking
    ├── dummy.rs                       ← no-op variant used by parachain nodes
    └── tests.rs
```

Of these, `lib.rs` is where everything important is. The other files are auxiliary.

## 5. The `#[orchestra]` Macro: Generated, Not Hand-Written

The Overseer struct itself is a **declarative skeleton** decorated with the `#[orchestra(...)]` procedural macro from the [`orchestra`](https://crates.io/crates/orchestra) crate (it's an external dependency, not part of polkadot-sdk):

**File:** `polkadot/node/overseer/src/lib.rs:476-483`

```rust
#[orchestra(
    gen=AllMessages,
    event=Event,
    signal=OverseerSignal,
    error=SubsystemError,
    message_capacity=2048,
)]
pub struct Overseer<SupportsParachains> {
    // ... fields ...
}
```

This macro reads the struct definition at compile time and generates:

- One mpsc channel per `#[subsystem]` field.
- A `route_message` function that pattern-matches the message type and delivers to the right channel.
- A `route_message_with_priority<P>` variant for high-priority messages (more on this in Section 14).
- The `broadcast_signal` function that fans an `OverseerSignal` out to every subsystem.
- A builder API with `replace_*` methods used by integration tests to swap subsystems.
- Implementations of the `SubsystemSender` and `SubsystemContext` traits that each subsystem uses to talk back.

If you want to see the expanded code, the `Cargo.toml` of `polkadot-overseer` exposes a feature flag:

```toml
expand = ["orchestra/expand"]
```

Enabling it makes the macro print the expanded source. Useful when debugging routing issues.

## 6. The Subsystem List

**File:** `polkadot/node/overseer/src/lib.rs:483-660`

Each field of the `Overseer` struct is a subsystem, and the `#[subsystem(...)]` attribute on it carries two pieces of information:

- The message type the subsystem **receives** (the first positional argument or by inference).
- A `sends:` list of message types the subsystem is allowed to **emit**.

A representative example:

```rust
#[subsystem(CandidateValidationMessage, sends: [
    ChainApiMessage,
    RuntimeApiMessage,
])]
candidate_validation: CandidateValidation,

#[subsystem(blocking, RuntimeApiMessage, sends: [])]
runtime_api: RuntimeApi,

#[subsystem(blocking, NetworkBridgeRxMessage, sends: [
    BitfieldDistributionMessage,
    StatementDistributionMessage,
    ApprovalVotingParallelMessage,
    GossipSupportMessage,
    DisputeDistributionMessage,
    CollationGenerationMessage,
    CollatorProtocolMessage,
])]
network_bridge_rx: NetworkBridgeRx,
```

A few attributes worth noting:

- **`blocking`** — tells the macro the subsystem may block, so it should be spawned via `spawn_blocking` (a dedicated thread pool that doesn't share with non-blocking async tasks). Used by `runtime_api`, `availability_store`, `network_bridge_rx`, etc.
- **`message_capacity: N`** — overrides the default channel capacity (2048) for this subsystem. `bitfield_distribution` uses 8192 because bitfields are bursty.
- **`can_receive_priority_messages`** — tells the macro to wire the high-priority variant of `route_message_with_priority`. Used by latency-critical subsystems like `statement_distribution`, `bitfield_distribution`, `dispute_coordinator`.

The full list (~26 subsystems as of v17 of the runtime API) covers everything from consensus (`approval_voting_parallel`) to storage (`availability_store`) to networking glue (`network_bridge_rx`, `network_bridge_tx`).

# Part III: The Substrate Client — The Other Side of the Bridge

## 7. What "Substrate Client" Means in This Context

When this document says "Substrate Client", it means the concrete `Arc<TFullClient<Block, RuntimeApi, Executor>>` from the `sc-service` crate. This is the lower-level layer of the node that:

- Owns the **block database** (RocksDB or ParityDB).
- Imports blocks from the network (validating them, executing the runtime).
- Executes the **runtime** (the WASM blob compiled from the chain's `runtime` crate).
- Exposes **notification streams**: `import_notification_stream()`, `finality_notification_stream()`, and others.

Polkadot doesn't write its own client. It instantiates the generic Substrate one and adds the validator-specific logic on top, via the Overseer + subsystems.

In Polkadot, the alias is:

**File:** `polkadot/node/service/src/lib.rs:105`

```rust
pub type FullClient = sc_service::TFullClient<...>;
```

## 8. Where the Client Is Instantiated

**File:** `polkadot/node/service/src/builder/partial.rs:91-145`

The function `new_partial_basics` constructs the four pieces every node needs:

```rust
pub(crate) fn new_partial_basics(
    config: &mut Configuration,
    telemetry_worker_handle: Option<TelemetryWorkerHandle>,
) -> Result<Basics, Error> {
    // ...
    let executor = WasmExecutor::builder()
        .with_execution_method(config.executor.wasm_method)
        .with_onchain_heap_alloc_strategy(heap_pages)
        .with_offchain_heap_alloc_strategy(heap_pages)
        .with_max_runtime_instances(config.executor.max_runtime_instances)
        .with_runtime_cache_size(config.executor.runtime_cache_size)
        .build();

    let (client, backend, keystore_container, task_manager) = sc_service::new_full_parts(
        &config,
        telemetry.as_ref().map(|(_, telemetry)| telemetry.handle()),
        executor,
        vec![Arc::new(GrandpaPruningFilter)],
    )?;
    let client = Arc::new(client);
    // ...
}
```

`sc_service::new_full_parts` is the Substrate library function that creates:

- `client: TFullClient<...>` — the Substrate Client itself.
- `backend: TFullBackend<...>` — the disk-backed database.
- `keystore_container: KeystoreContainer` — the on-disk key storage.
- `task_manager: TaskManager` — the spawner Polkadot uses for all its background tasks.

These are returned wrapped in a `Basics` struct (`partial.rs:81-89`) that downstream code unpacks.

## 9. The Streams the Client Exposes

The Client implements the trait `BlockchainEvents<Block>` (from `sc-client-api`). Two methods on this trait are relevant for the Overseer:

```rust
trait BlockchainEvents<B: BlockT> {
    fn import_notification_stream(&self) -> ImportNotifications<B>;
    fn finality_notification_stream(&self) -> FinalityNotifications<B>;
    // ... other streams ...
}
```

These streams emit notifications:

- **`import_notification_stream()`** — emits a `BlockImportNotification` every time a block is imported into the database (whether from network gossip, sync, or local production).
- **`finality_notification_stream()`** — emits a `FinalityNotification` every time a block is finalized by GRANDPA (or whichever finality gadget the chain uses).

The Overseer doesn't subscribe to these streams directly. A glue function called `forward_events` does, and forwards events to the Overseer's public `Handle` (Section 12).

# Part IV: Wiring the Overseer to the Client

## 10. The Service Builder: Where It All Comes Together

**File:** `polkadot/node/service/src/builder/mod.rs:194` (`PolkadotServiceBuilder::build`)

This single function is where the Client and the Overseer become connected. It runs once during node startup, in three relevant steps for our purposes (a, b, c below).

## 11. The Client Goes Two Places

**File:** `polkadot/node/service/src/builder/mod.rs:569`

The `Arc<FullClient>` that came out of `new_partial_basics` is cloned (cheap — `Arc::clone` only bumps a refcount) and stored in `overseer_client`:

```rust

That clone is then **used twice**:

### (a) As the runtime backend for the `runtime-api` subsystem

```

`DefaultSubsystemClient` is the wrapper the `runtime-api` subsystem uses to call into the runtime. When some other subsystem sends `RuntimeApiMessage::Request(relay_parent, RuntimeApiRequest::SessionExecutorParams(...))` through the Overseer, it lands in `runtime-api`, which calls `runtime_client.runtime_api().session_executor_params(at, idx).await`. That call goes through the `Arc<FullClient>` we cloned here.

The `OverseerGenArgs` struct also carries other dependencies the macro-generated code wires into specific subsystems:

- `network_service`, `sync_service` → consumed by `network-bridge` and `gossip-support`.
- `collation_req_*_receiver` → mpsc receivers for incoming collator requests, consumed by `collator-protocol`.
- `available_data_req_receiver` → for `availability-distribution`.
- `notification_services` → libp2p notification protocol handles, also for `network-bridge`.

These are the **network channels** that subsystems use. Note that they don't go through the Overseer either: the Overseer only routes inter-subsystem messages. Network I/O is the subsystem's own business.

### (b) As the source of block-import notifications

**File:** `polkadot/node/service/src/builder/mod.rs:656-676`

```rust
{
    let handle = handle.clone();
    task_manager.spawn_essential_handle().spawn_blocking(
        "overseer",
        None,
        Box::pin(async move {
            use futures::{pin_mut, select, FutureExt};

            let forward = polkadot_overseer::forward_events(overseer_client, handle);
            //                                              ^^^^^^^^^^^^^^^  ^^^^^^
            //                                              the same client  the Handle

            let forward = forward.fuse();
            let overseer_fut = overseer.run().fuse();

            pin_mut!(overseer_fut);
            pin_mut!(forward);

            select! {
                () = forward => (),         // pumps client events → Overseer
                () = overseer_fut => (),    // runs the Overseer's main loop
                complete => (),
            }
        }),
    );
}
```

A single task named `"overseer"` is spawned via `spawn_essential_handle().spawn_blocking()`. It hosts **both** the Overseer's main loop and the `forward_events` pump, multiplexed by `select!`. If either future returns (which only happens on shutdown), the task ends. Because it's an *essential* task, the whole node shuts down with it.

## 12. `forward_events`: The Pump from Client to Overseer

**File:** `polkadot/node/overseer/src/lib.rs:332-359`

```rust
/// Glues together the [`Overseer`] and `BlockchainEvents` by forwarding
/// import and finality notifications into the [`OverseerHandle`].
pub async fn forward_events<P: BlockchainEvents<Block>>(client: Arc<P>, mut handle: Handle) {
    let mut finality = client.finality_notification_stream();
    let mut imports = client.import_notification_stream();

    loop {
        select! {
            f = finality.next() => {
                match f {
                    Some(block) => {
                        handle.block_finalized(block.into()).await;
                    }
                    None => break,
                }
            },
            i = imports.next() => {
                match i {
                    Some(block) => {
                        handle.block_imported(block.into()).await;
                    }
                    None => break,
                }
            },
            complete => break,
        }
    }
}
```

This is the only piece of code that connects the two sides. It is **infinitely** simple in shape:

1. Subscribe to both notification streams from the Client.
2. Loop forever, multiplexing the two with `select!`.
3. For each event, call the corresponding method on the Overseer's `Handle`.

The `block.into()` conversion translates Substrate's `BlockImportNotification` into the Overseer's local `BlockInfo` type (`overseer/src/lib.rs:280-289`).

## 13. `Handle` and `OverseerHandle`: The Public Surface

**File:** `polkadot/node/overseer/src/lib.rs:180-255`

The Overseer itself is wrapped twice:

- `OverseerHandle` — generated by the `#[orchestra]` macro. A typed sender into the Overseer's events channel.
- `Handle` (`pub struct Handle(OverseerHandle)`) — a thin newtype around `OverseerHandle` that exposes hand-written async methods like `block_imported`, `block_finalized`, `send_msg`, `wait_for_activation`, `stop`. It's the API the rest of the node uses.

The relevant methods:

```rust
pub async fn block_imported(&mut self, block: BlockInfo) {
    self.send_and_log_error(Event::BlockImported(block)).await
}

pub async fn block_finalized(&mut self, block: BlockInfo) {
    self.send_and_log_error(Event::BlockFinalized(block)).await
}

pub async fn send_msg(&mut self, msg: impl Into<AllMessages>, origin: &'static str) {
    self.send_msg_with_priority(msg, origin, PriorityLevel::Normal).await
}
```

Each of these turns its argument into an `Event` variant (Section 15) and pushes it onto the Overseer's events channel. The Overseer's main loop pulls it off the other end.

# Part V: Inside the Overseer Loop

## 14. The Three Channels of `run_inner`

**File:** `polkadot/node/overseer/src/lib.rs:771-826`

The main loop is a single `select!` that waits on three event sources:

```rust
async fn run_inner(mut self) -> SubsystemResult<()> {
    let metrics = self.metrics.clone();
    spawn_metronome_metrics(&mut self, metrics)?;

    loop {
        select! {
            msg = self.events_rx.select_next_some() => {
                // ↑ events from the outside world (Handle::send_msg, forward_events, etc.)
                match msg {
                    Event::MsgToSubsystem { msg, origin, priority } => {
                        match priority {
                            PriorityLevel::Normal => {
                                self.route_message(msg.into(), origin).await?;
                            },
                            PriorityLevel::High => {
                                self.route_message_with_priority::<HighPriority>(msg.into(), origin).await?;
                            },
                        }
                        self.metrics.on_message_relayed();
                    }
                    Event::Stop => {
                        self.stop().await;
                        return Ok(());
                    }
                    Event::BlockImported(block) => {
                        self.block_imported(block).await?;
                    }
                    Event::BlockFinalized(block) => {
                        self.block_finalized(block).await?;
                    }
                    Event::ExternalRequest(request) => {
                        self.handle_external_request(request);
                    }
                }
            },
            msg = self.to_orchestra_rx.select_next_some() => {
                // ↑ requests *from* subsystems back to the Overseer
                match msg {
                    ToOrchestra::SpawnJob { name, subsystem, s } => {
                        self.spawn_job(name, subsystem, s);
                    }
                    ToOrchestra::SpawnBlockingJob { name, subsystem, s } => {
                        self.spawn_blocking_job(name, subsystem, s);
                    }
                }
            },
            res = self.running_subsystems.select_next_some() => {
                // ↑ a subsystem task finished — either cleanly or with an error
                gum::error!(
                    target: LOG_TARGET,
                    subsystem = ?res,
                    "subsystem finished unexpectedly",
                );
                self.stop().await;
                return res;
            },
        }
    }
}
```

The three sources:

1. **`events_rx`** — receives `Event` values. This is the channel `Handle` writes to. In production, almost everything except internal subsystem-to-Overseer requests comes through here.
2. **`to_orchestra_rx`** — internal channel for **subsystems calling back** to the Overseer. The macro hands subsystems a `ChannelsOut` they can use to ask the Overseer to spawn a job on their behalf.
3. **`running_subsystems`** — a `FuturesUnordered` of all subsystem tasks. If any of them finishes (normally or with an error), the Overseer treats that as a fatal event and shuts down.

## 15. The Five Event Variants

**File:** `polkadot/node/overseer/src/lib.rs:294-318`

```rust
pub enum Event {
    BlockImported(BlockInfo),
    BlockFinalized(BlockInfo),
    MsgToSubsystem {
        msg: AllMessages,
        origin: &'static str,
        priority: PriorityLevel,
    },
    ExternalRequest(ExternalRequest),
    Stop,
}
```

Each variant is dispatched in the `match` of `run_inner`:

| Variant | Source | Effect |
|---|---|---|
| `BlockImported` | `forward_events` (from `import_notification_stream`) | `block_imported(...)` → maybe `broadcast_signal(ActiveLeaves)` |
| `BlockFinalized` | `forward_events` (from `finality_notification_stream`) | `block_finalized(...)` → `broadcast_signal(BlockFinalized)`, possibly also an `ActiveLeaves` for deactivations |
| `MsgToSubsystem` | `Handle::send_msg(...)` from inside or outside the node | Routed to one specific subsystem |
| `ExternalRequest` | `Handle::wait_for_activation(...)` and similar | Stores a oneshot sender; replied to when the leaf activates |
| `Stop` | `Handle::stop().await` | Triggers `stop()` and returns |

`MsgToSubsystem` is the only one with a `priority` field. The macro generates two routing functions; the priority variant uses a different per-channel send strategy that allows the message to jump ahead. Only subsystems annotated with `can_receive_priority_messages` accept these.

## 16. From `BlockImported` to `OverseerSignal::ActiveLeaves`

This is the most important transformation in the Overseer for understanding any flow that reacts to block production.

**File:** `polkadot/node/overseer/src/lib.rs:828-858`

```rust
async fn block_imported(&mut self, block: BlockInfo) -> SubsystemResult<()> {
    match self.active_leaves.entry(block.hash) {
        hash_map::Entry::Vacant(entry) => entry.insert(block.number),
        hash_map::Entry::Occupied(entry) => {
            debug_assert_eq!(*entry.get(), block.number);
            return Ok(());
        },
    };

    let mut update = match self.on_head_activated(&block.hash, Some(block.parent_hash)).await {
        Some(_) => ActiveLeavesUpdate::start_work(ActivatedLeaf {
            hash: block.hash,
            number: block.number,
            unpin_handle: block.unpin_handle,
        }),
        None => ActiveLeavesUpdate::default(),
    };

    if let Some(number) = self.active_leaves.remove(&block.parent_hash) {
        debug_assert_eq!(block.number.saturating_sub(1), number);
        update.deactivated.push(block.parent_hash);
        self.on_head_deactivated(&block.parent_hash);
    }

    self.clean_up_external_listeners();

    if !update.is_empty() {
        self.broadcast_signal(OverseerSignal::ActiveLeaves(update)).await?;
    }
    Ok(())
}
```

Step by step:

1. Insert the new block into `active_leaves`. If it was already known, drop the event.
2. Call `on_head_activated`, which checks whether the block's state actually supports the parachain runtime API (`SupportsParachains` trait). If not (this can happen for very old blocks), the activation is suppressed.
3. Build an `ActiveLeavesUpdate` with the new leaf in `activated`.
4. If the parent was an active leaf (it almost always is when a normal block is imported), remove it from `active_leaves` and add it to `deactivated`. This way subsystems that were doing work on the parent know it's no longer the cabeza.
5. Notify any external `wait_for_activation` listeners that the leaf is now live.
6. Broadcast `OverseerSignal::ActiveLeaves(update)` to every subsystem.

The end result: every imported block that becomes a fork tip turns into a single `ActiveLeaves` signal that fans out to all subsystems.

## 17. Tracking Active Leaves

The Overseer keeps a `HashMap<Hash, BlockNumber>` of currently-active leaves:

**File:** `polkadot/node/overseer/src/lib.rs:666`

```rust
/// The set of the "active leaves".
pub active_leaves: HashMap<Hash, BlockNumber>,
```

A "leaf" is a block that is currently a tip of some active fork the node cares about. When a child of an existing leaf is imported, the parent stops being a leaf (it's now an internal node) and the child takes its place. When a block is finalized, all orphaned leaves whose number is at or below the finalized block are pruned.

The pruning logic is in `block_finalized`:

**File:** `polkadot/node/overseer/src/lib.rs:860-890`

```rust
async fn block_finalized(&mut self, block: BlockInfo) -> SubsystemResult<()> {
    let mut update = ActiveLeavesUpdate::default();

    self.active_leaves.retain(|h, n| {
        if *n <= block.number && *h != block.hash {
            update.deactivated.push(*h);
            false
        } else {
            true
        }
    });

    for deactivated in &update.deactivated {
        self.on_head_deactivated(deactivated)
    }

    self.broadcast_signal(OverseerSignal::BlockFinalized(block.hash, block.number)).await?;

    if !update.is_empty() {
        self.broadcast_signal(OverseerSignal::ActiveLeaves(update)).await?;
    }
    Ok(())
}
```

`BlockFinalized` is broadcast first, then a separate `ActiveLeaves` carrying only `deactivated` entries is broadcast for each orphaned leaf.

## 18. `broadcast_signal`: One Message, All Subsystems

`broadcast_signal` is generated by the `#[orchestra]` macro. Conceptually it does:

```rust
async fn broadcast_signal(&mut self, signal: OverseerSignal) -> SubsystemResult<()> {
    for sender in self.signal_senders.iter_mut() {
        sender.send_signal(signal.clone()).await?;
    }
    Ok(())
}
```

In practice the macro generates one `send` per declared subsystem field, calling each subsystem's signal sender directly. Signals go through a separate channel from messages — they have higher priority and are delivered ahead of any pending `MsgToSubsystem`.

This "signals always preempt messages" property is what lets a subsystem reliably react to leaf changes even when it has a backlog of work.

There are exactly three signal variants:

**File:** `polkadot/node/subsystem-types/src/lib.rs:110-119`

```rust
pub enum OverseerSignal {
    /// Subsystems should adjust their jobs to start and stop work on appropriate block hashes.
    ActiveLeaves(ActiveLeavesUpdate),
    /// `Subsystem` is informed of a finalized block by its block hash and number.
    BlockFinalized(Hash, BlockNumber),
    /// Conclude the work of the `Overseer` and all `Subsystem`s.
    Conclude,
}
```

`Conclude` is sent by `stop()` during shutdown and tells every subsystem to drain and exit.

# Part VI: The Subsystem Side of the Wire

## 19. How a Subsystem Receives Signals

A subsystem is just an async function that runs forever, blocked on its incoming channel. The macro hands it a `Context` that exposes `recv()`, which yields whatever was sent through any of the channels (signals or messages).

A real example, using the candidate-validation subsystem:

**File:** `polkadot/node/core/candidate-validation/src/lib.rs:560-610` (abbreviated)

```rust
loop {
    futures::select! {
        comm = ctx.recv().fuse() => {
            match comm {
                Ok(FromOrchestra::Signal(OverseerSignal::ActiveLeaves(update))) => {
                    handle_active_leaves_update(
                        ctx.sender(),
                        keystore.clone(),
                        &mut validation_host,
                        update,
                        &mut state,
                    ).await
                },
                Ok(FromOrchestra::Signal(OverseerSignal::BlockFinalized(..))) => {},
                Ok(FromOrchestra::Signal(OverseerSignal::Conclude)) => return Ok(()),
                Ok(FromOrchestra::Communication { msg }) => {
                    // ... handle subsystem-specific messages ...
                },
                // ...
            }
        }
    }
}
```

`FromOrchestra` is the enum the macro generates to carry both kinds of incoming traffic:

- `FromOrchestra::Signal(OverseerSignal)` — a global signal from the Overseer.
- `FromOrchestra::Communication { msg }` — a typed message addressed specifically to this subsystem.

## 20. Signals vs Communication: Two Kinds of Traffic

The distinction matters for understanding latency and ordering:

| Aspect | `Signal` | `Communication` |
|---|---|---|
| Sender | The Overseer itself | Any other subsystem (or an external `Handle::send_msg`) |
| Cardinality | Broadcast to **all** subsystems | Routed to **one** specific subsystem |
| Variants | Three: `ActiveLeaves`, `BlockFinalized`, `Conclude` | One per `RuntimeApiRequest`/`CandidateValidationMessage`/etc. — hundreds total |
| Channel | Dedicated signal channel per subsystem | Shared message channel per subsystem (with priority lane for some) |
| Preemption | Always preempts pending messages | Goes into FIFO order on the message channel |

A subsystem's `recv()` interleaves both, but signals tend to win on contention because the macro implements them on a separate priority channel.

## 21. Cross-Subsystem Messaging Through the Overseer

When subsystem A wants subsystem B to do something, A doesn't have a reference to B. Instead, A's `Context` has a `sender` of type `impl SubsystemSender<MessageType>`, and A pushes a message into it. The macro's wiring delivers it to B's incoming channel through the Overseer's shared message bus.

A typical example from the PR we used as a case study:

**File:** `polkadot/node/subsystem-util/src/lib.rs:454-500` (abbreviated)

```rust
pub async fn executor_params_at_relay_parent(
    relay_parent: Hash,
    sender: &mut impl overseer::SubsystemSender<RuntimeApiMessage>,
) -> Result<ExecutorParams, Error> {
    match request_session_index_for_child(relay_parent, sender).await.await {
        // ...
        Ok(Ok(session_index)) => {
            match request_session_executor_params(relay_parent, session_index, sender).await.await {
                // ...
            }
        },
    }
}
```

The pattern:

1. Caller creates a `oneshot::channel::<Result<T, _>>()`.
2. Caller wraps the sender half in a `RuntimeApiRequest` variant and pushes the whole thing as `RuntimeApiMessage::Request(relay_parent, request)` into its `SubsystemSender<RuntimeApiMessage>`.
3. The Overseer routes the `RuntimeApiMessage` to the `runtime-api` subsystem.
4. `runtime-api` resolves the request (calling the runtime through the `Arc<FullClient>` it received in Section 11) and sends the result through the oneshot's sender half.
5. Caller awaits the receiver half.

The double `.await.await` you see in the snippet (`request_session_index_for_child(...).await.await`) is exactly this: the first await sends the message, the second one awaits the oneshot reply.

## 22. The Setup: A Validator About to Rotate In

The candidate-validation subsystem prepares parachain WASM (`Parachain Validation Function` = PVF) ahead of time, so that when the node becomes a validator next session, the artifacts are already cached on disk. ## 26. Why the Overseer Is the Right Abstraction Here

It's worth stepping back. What did the Overseer actually *do* in this flow?

- It turned a single Substrate-level `BlockImportNotification` into 26 parallel `ActiveLeaves` deliveries — one per subsystem — without any of the subsystems knowing about each other.
- It routed every cross-subsystem message (`CandidateEvents`, `SessionExecutorParams`, etc.) without either side holding a reference to the other.
- It did this with type safety: the macro refuses to compile a subsystem that emits a message it didn't declare in `sends:`.
- It did this with backpressure: each subsystem has its own bounded channel and the priority lane lets latency-sensitive messages jump.

If we removed the Overseer and replaced it with direct calls between subsystems, the dependency graph (`#[subsystem(... sends: [...])]` lines in `lib.rs:483-660`) would become a tangle of trait objects, and the whole "broadcast `ActiveLeaves` to everyone interested" pattern would have to be re-implemented from scratch by every emitter. The Overseer is what keeps that complexity in one place.

# Part VIII: Operational Considerations

## 27. The Overseer as an Essential Task

**File:** `polkadot/node/service/src/builder/mod.rs:656`

```rust
task_manager.spawn_essential_handle().spawn_blocking(
    "overseer",
    None,
    /* future */,
);
```

`spawn_essential_handle` (Substrate's `TaskManager` API) marks the task as critical: if it panics or exits, the entire node shuts down. There is no graceful degradation. Without the Overseer, no subsystem can communicate, so the node is useless.

`spawn_blocking` puts it on the dedicated blocking thread pool. This is because a few of the Overseer's internal operations (like `select!`-ing across many channels under heavy load) can briefly block, and we don't want that to starve other async tasks.

## 28. Channel Capacities and Backpressure

Two knobs in the macro decide how messages queue up under load:

- The default `message_capacity = 2048` in `#[orchestra(...)]`.
- The per-subsystem `message_capacity: N` override in `#[subsystem(...)]`.

When a sender calls `.send_message(msg).await` and the destination's channel is full, the sender backpressures — its `await` doesn't resolve until the destination drains. In practice this means a slow subsystem can throttle its callers, which is mostly desirable; the alternative would be unbounded queues that grow until OOM.

The exception is the priority lane (`can_receive_priority_messages`). High-priority messages have their own queue with separate capacity, so latency-critical paths (statement distribution, dispute coordination) aren't blocked by ordinary work.

## 29. Signal Ordering and Determinism

One subtle property: signals are delivered via a separate per-subsystem channel from messages, and the macro's generated code picks signals first when `recv()` has both available. This means a subsystem that has 1000 backlogged messages and just received an `ActiveLeaves` will process the `ActiveLeaves` next, not after draining the backlog.

This matters for correctness: a subsystem like `candidate-validation` keeps state keyed by the most recent leaf hash. If the leaf changed, processing a stale message (one that referenced a deactivated block) wastes work or worse, gives a wrong answer.

Within a single category (signals or messages), ordering is FIFO per channel. There's no global ordering across subsystems — two subsystems may observe different orderings of independent events. In practice the only ordering guarantee that matters is local to one subsystem's incoming channels.
